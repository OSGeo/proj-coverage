       7             : ********************************************************************************
       8             : 
       9             :     Geodetic transformations are typically organized in a number of
      10             :     steps. For example, a datum shift could be carried out through
      11             :     these steps:
      12             : 
      13             :     1. Convert (latitude, longitude, ellipsoidal height) to
      14             :        3D geocentric cartesian coordinates (X, Y, Z)
      15             :     2. Transform the (X, Y, Z) coordinates to the new datum, using a
      16             :        7 parameter Helmert transformation.
      17             :     3. Convert (X, Y, Z) back to (latitude, longitude, ellipsoidal height)
      18             : 
      19             :     If the height system used is orthometric, rather than ellipsoidal,
      20             :     another step is needed at each end of the process:
      21             : 
      22             :     1. Add the local geoid undulation (N) to the orthometric height
      23             :        to obtain the ellipsoidal (i.e. geometric) height.
      24             :     2. Convert (latitude, longitude, ellipsoidal height) to
      25             :        3D geocentric cartesian coordinates (X, Y, Z)
      26             :     3. Transform the (X, Y, Z) coordinates to the new datum, using a
      27             :        7 parameter Helmert transformation.
      28             :     4. Convert (X, Y, Z) back to (latitude, longitude, ellipsoidal height)
      29             :     5. Subtract the local geoid undulation (N) from the ellipsoidal height
      30             :        to obtain the orthometric height.
      31             : 
      32             :     Additional steps can be added for e.g. change of vertical datum, so the
      33             :     list can grow fairly long. None of the steps are, however, particularly
      34             :     complex, and data flow is strictly from top to bottom.
      35             : 
      36             :     Hence, in principle, the first example above could be implemented using
      37             :     Unix pipelines:
      38             : 
      39             :     cat my_coordinates | geographic_to_xyz | helmert | xyz_to_geographic > my_transformed_coordinates
      40             : 
      41             :     in the grand tradition of Software Tools [1].
      42             : 
      43             :     The proj pipeline driver implements a similar concept: Stringing together
      44             :     a number of steps, feeding the output of one step to the input of the next.
      45             : 
      46             :     It is a very powerful concept, that increases the range of relevance of the
      47             :     proj.4 system substantially. It is, however, not a particularly intrusive
      48             :     addition to the PROJ.4 code base: The implementation is by and large completed
      49             :     by adding an extra projection called "pipeline" (i.e. this file), which
      50             :     handles all business, and a small amount of added functionality in the
      51             :     pj_init code, implementing support for multilevel, embedded pipelines.
      52             : 
      53             :     Syntactically, the pipeline system introduces the "+step" keyword (which
      54             :     indicates the start of each transformation step), and reintroduces the +inv
      55             :     keyword (indicating that a given transformation step should run in reverse, i.e.
      56             :     forward, when the pipeline is executed in inverse direction, and vice versa).
      57             : 
      58             :     Hence, the first transformation example above, can be implemented as:
      59             : 
      60             :     +proj=pipeline +step proj=cart +step proj=helmert <ARGS> +step proj=cart +inv
      61             : 
      62             :     Where <ARGS> indicate the Helmert arguments: 3 translations (+x=..., +y=...,
      63             :     +z=...), 3 rotations (+rx=..., +ry=..., +rz=...) and a scale factor (+s=...).
      64             :     Following geodetic conventions, the rotations are given in arcseconds,
      65             :     and the scale factor is given as parts-per-million.
      66             :
