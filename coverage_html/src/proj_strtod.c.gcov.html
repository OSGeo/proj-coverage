       7             : ************************************************************************
       8             : 
       9             : Conventionally, PROJ.4 does not honor locale settings, consistently
      10             : behaving as if LC_ALL=C.
      11             : 
      12             : For this to work, we have, for many years, been using other solutions
      13             : than the C standard library strtod/atof functions for converting strings
      14             : to doubles.
      15             : 
      16             : In the early versions of proj, iirc, a gnu version of strtod was used,
      17             : mostly to work around cases where the same system library was used for
      18             : C and Fortran linking, hence making strtod accept "D" and "d" as
      19             : exponentiation indicators, following Fortran Double Precision constant
      20             : syntax. This broke the proj angular syntax, accepting a "d" to mean
      21             : "degree": 12d34'56", meaning 12 degrees 34 minutes and 56 seconds.
      22             : 
      23             : With an explicit MIT licence, PROJ.4 could not include GPL code any
      24             : longer, and apparently at some time, the GPL code was replaced by the
      25             : current C port of a GDAL function (in pj_strtod.c), which reads the
      26             : LC_NUMERIC setting and, behind the back of the user, momentarily changes
      27             : the conventional '.' delimiter to whatever the locale requires, then
      28             : calls the system supplied strtod.
      29             : 
      30             : While this requires a minimum amount of coding, it only solves one
      31             : problem, and not in a very generic way.
      32             : 
      33             : Another problem, I would like to see solved, is the handling of underscores
      34             : as generic delimiters. This is getting popular in a number of programming
      35             : languages (Ada, C++, C#, D, Java, Julia, Perl 5, Python, Rust, etc.
      36             : cf. e.g. https://www.python.org/dev/peps/pep-0515/), and in our case of
      37             : handling numbers being in the order of magnitude of the Earth's dimensions,
      38             : and a resolution of submillimetre, i.e. having 10 or more significant digits,
      39             : splitting the "wall of digits" into smaller chunks is of immense value.
      40             : 
      41             : Hence this reimplementation of strtod, which hardcodes '.' as indicator of
      42             : numeric fractions, and accepts '_' anywhere in a numerical string sequence:
      43             : So a typical northing value can be written
      44             : 
      45             :                             6_098_907.8250 m
      46             : rather than
      47             :                              6098907.8250 m
      48             : 
      49             : which, in my humble opinion, is well worth the effort.
      50             : 
      51             : While writing this code, I took ample inspiration from Michael Ringgaard's
      52             : strtod version over at http://www.jbox.dk/sanos/source/lib/strtod.c.html,
      53             : and Yasuhiro Matsumoto's public domain version over at
      54             : https://gist.github.com/mattn/1890186. The code below is, however, not
      55             : copied from any of the two mentioned - it is a reimplementation, and
      56             : probably suffers from its own set of bugs. So for now, it is intended
      57             : not as a replacement of pj_strtod, but only as an experimental piece of
      58             : code for use in an experimental new transformation program, cct.
